Open Core vs Hosted-Only: How do you monetize?

CloudMock is source-available (BSL). The SaaS platform needs a clear boundary between what's free and what's paid.

A

Open Core (Public Repo + Proprietary SaaS)

CloudMock stays public and free. The SaaS dashboard, multi-tenant API, HIPAA controls, and team features live in a private repo. Users can self-host CloudMock but can't self-host the platform.

How others do it:
GitLab CE (free) vs GitLab EE (paid). Supabase (open) vs Supabase Cloud (hosted). PostHog (open) vs PostHog Cloud.

Pros

  • Clear monetization boundary
  • Open source drives adoption
  • SaaS features stay proprietary
  • No licensing complexity

Cons

  • Two repos to manage
  • Community can't contribute to SaaS features
B

Source-Available Everything (BSL for SaaS too)

SaaS code lives in the same public repo under BSL. Anyone can read it, run it locally, even self-host internally -- but can't offer it as a competing hosted service. You sell the hosted version as the convenience layer.

How others do it:
HashiCorp (BSL), Sentry (BSL), CockroachDB (BSL). All source-available, monetize the hosted version.

Pros

  • One repo, simpler management
  • Transparency builds trust (esp. for HIPAA)
  • Community can audit security code
  • Self-hosting allowed = more adoption

Cons

  • Competitors can read your SaaS code
  • Self-hosters don't pay you
  • BSL enforcement is untested legally
C

Hybrid: Open Emulator + Gated Enterprise Features

CloudMock core stays public (BSL). Enterprise/SaaS features (HIPAA, audit logs, SSO, multi-tenant API) are feature-gated behind a license key in the same repo. Code is visible but requires a paid license to activate in production.

How others do it:
Elasticsearch (open + X-Pack). GitLab (CE + EE in same repo, license-gated). Airbyte (open + enterprise connectors).

Pros

  • One repo, one codebase
  • Self-hosting allowed with license
  • Enterprise features visible (trust) but gated (revenue)
  • Upsell path: free → self-host → hosted

Cons

  • License key infra to build
  • Gate bypass is trivial (code is visible)
  • More complex build/release process

Key insight: You already chose BSL which converts to Apache 2.0 in 2030. The question is whether SaaS/enterprise features get the same treatment or stay proprietary forever. HIPAA customers generally prefer being able to audit the code, which favors B or C.
