Source Integration

How do Flutter, React Native, Web, and Node apps connect to the devtools?

SDK Approach

How Sources Connect

@cloudmock/web
import { CloudMock } from '@cloudmock/web'
▼ WebSocket to devtools ▼
Devtools App

What the SDKs Capture

Network Requests
HTTP interceptor captures all outgoing requests — URL, method, headers, body, timing, status. Automatically correlates with cloudmock's server-side log via request ID header injection.
Console Logs
Intercepts print/console.log/debugPrint with level, timestamp, and source location (file:line). Platform-native: Dart zones for Flutter, LogBox for RN, console API for web.
Errors & Crashes
Uncaught exceptions with stack traces. Flutter: FlutterError.onError + PlatformDispatcher. RN: ErrorUtils. Web: window.onerror + unhandledrejection.
Performance
App launch time, frame rate (Flutter/RN), memory usage, navigation timing. Feeds into Metrics and SLO views.

Zero-Config Integration

Dev-only, tree-shaken in production. SDKs wrap in kDebugMode / __DEV__ / process.env.NODE_ENV checks. Zero overhead in release builds — no need to remove imports.
Auto-discovery. SDKs find the devtools app via mDNS/Bonjour on the local network. No hardcoded ports or config. Works with simulators, emulators, and physical devices on the same WiFi.
Graceful when devtools isn't running. If the devtools app isn't open, the SDK silently no-ops. No crashes, no warnings, no performance impact.
cloudmock correlation is automatic. The SDK injects an X-CloudMock-Source header on HTTP requests. Cloudmock reads it and tags the server-side log entry. The devtools links both sides by request ID.

Alternative: No-SDK Mode

For apps that can't add an SDK, the devtools can also act as a proxy. Point your app's API base URL at the devtools (e.g., localhost:4580) instead of cloudmock directly. The devtools forwards to cloudmock and captures the traffic — like Charles Proxy but built into the tool. Less data (no console logs or crashes), but zero app changes needed.
